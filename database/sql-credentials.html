Key Points:
🗝️ Primary Key: StudentID - Uniquely identifies each student
🔄 Alternate Key: AadharNo - Unique but not chosen as Primary
🎯 Candidate Keys: StudentID, Email, AadharNo - All can uniquely identify records

📚 Enrollment Table Example

🧩 StudentID (CK) 🧩 SubjectCode (CK) Grade Semester
101 MATH101 A 1
101 PHY101 B+ 1
102 MATH101 A+ 1
Key Points:
🧩 Composite Key: (StudentID + SubjectCode) - Together uniquely identify records
🔗 Foreign Key: StudentID - Links to Students table’s Primary Key
🦸 Super Key: (StudentID + SubjectCode + Grade) - Includes extra columns but still unique

🔗 Table Relationships Visualization

/* Students Table */
CREATE TABLE Students (
StudentID INT PRIMARY KEY,
Name VARCHAR(50),
Email VARCHAR(100) UNIQUE,
AadharNo VARCHAR(12) UNIQUE
);
/* Enrollment Table */
CREATE TABLE Enrollment (
StudentID INT,
SubjectCode VARCHAR(10),
Grade VARCHAR(5),
PRIMARY KEY (StudentID, SubjectCode),
FOREIGN KEY (StudentID) REFERENCES Students(StudentID)
);

🔍 SQL Keys Comparison

Key Type Definition Properties Fun Analogy
🗝️ Primary Key Uniquely identifies each record Not NULL, Unique, Only one per table Like the head of a community—only one leader!
🔗 Foreign Key Links a column to a Primary Key in another table Can be NULL, Maintains referential integrity Like a connection to a trusted friend!
🎯 Candidate Key Any column(s) that can uniquely identify records Multiple per table, Unique, Minimal Like candidates eligible to lead the community!
🧩 Composite Key Combination of two or more columns Unique together, May not be unique individually Like a team effort—both members needed!
🦸 Super Key Any combination that uniquely identifies a row May include extra columns, Not minimal Like identifying someone with extra details!
🔄 Alternate Key Candidate Key not chosen as Primary Unique but not selected as Primary Like a backup leader who didn’t get chosen!
